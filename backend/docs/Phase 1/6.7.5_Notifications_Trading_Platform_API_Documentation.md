# 6.7.1 Notifications API Documentation

This API handles user notifications in the WasteTrade platform.

## API Endpoints

### Create Notification
```
POST /notifications
```

Creates a new notification for a user.

#### Request Body
```json
{
  "userId": 123,
  "type": "offer_received",
  "content": "You have received a new offer for your listing",
  "metadata": {
    "listingId": 456,
    "offerId": 789
  },
  "isRead": false
}
```

#### cURL Example
```bash
curl -L -X POST 'https://wastetrade-api-dev.b13devops.com/notifications' \
-H 'Authorization: Bearer YOUR_JWT_TOKEN' \
-H 'Content-Type: application/json' \
-d '{
  "userId": 123,
  "type": "offer_received", 
  "content": "You have received a new offer for your listing",
  "metadata": "{\"listingId\": 456, \"offerId\": 789}",
  "isRead": false
}'
```

#### Response
```json
{
  "id": 1,
  "userId": 123,
  "type": "offer_received",
  "content": "You have received a new offer for your listing",
  "metadata": "{\"listingId\": 456, \"offerId\": 789}",
  "isRead": false,
  "readAt": null,
  "createdAt": "2025-01-16T10:00:00.000Z"
}
```

### Get All Notifications
```
GET /notifications
```

Retrieves all notifications with optional filtering.

#### Query Parameters
- `filter` - JSON object for filtering notifications

#### cURL Example
```bash
curl -L -X GET 'https://wastetrade-api-dev.b13devops.com/notifications' \
-H 'Authorization: Bearer YOUR_JWT_TOKEN'
```

#### Filter Examples
```bash
# Get unread notifications only
curl -L -X GET 'https://wastetrade-api-dev.b13devops.com/notifications?filter=%7B%22where%22%3A%7B%22isRead%22%3Afalse%7D%7D' \
-H 'Authorization: Bearer YOUR_JWT_TOKEN'

# Get notifications for specific user
curl -L -X GET 'https://wastetrade-api-dev.b13devops.com/notifications?filter=%7B%22where%22%3A%7B%22userId%22%3A123%7D%7D' \
-H 'Authorization: Bearer YOUR_JWT_TOKEN'

# Get notifications by type
curl -L -X GET 'https://wastetrade-api-dev.b13devops.com/notifications?filter=%7B%22where%22%3A%7B%22type%22%3A%22offer_received%22%7D%7D' \
-H 'Authorization: Bearer YOUR_JWT_TOKEN'
```

#### Response
```json
[
  {
    "id": 1,
    "userId": 123,
    "type": "offer_received",
    "content": "You have received a new offer for your listing",
    "metadata": "{\"listingId\": 456, \"offerId\": 789}",
    "isRead": false,
    "readAt": null,
    "createdAt": "2025-01-16T10:00:00.000Z"
  }
]
```

### Update All Notifications
```
PATCH /notifications
```

Updates multiple notifications at once based on filter criteria.

#### Request Body
```json
{
  "isRead": true,
  "readAt": "2025-01-16T10:30:00.000Z"
}
```

#### cURL Example
```bash
# Mark all notifications as read for a user
curl -L -X PATCH 'https://wastetrade-api-dev.b13devops.com/notifications?where=%7B%22userId%22%3A123%7D' \
-H 'Authorization: Bearer YOUR_JWT_TOKEN' \
-H 'Content-Type: application/json' \
-d '{
  "isRead": true,
  "readAt": "2025-01-16T10:30:00.000Z"
}'
```

#### Response
```json
{
  "count": 3
}
```

## Notification Types

The following notification types are commonly used:

- `offer_received` - New offer received on a listing
- `offer_accepted` - Offer has been accepted
- `offer_rejected` - Offer has been rejected
- `offer_counter` - Counter offer received
- `listing_approved` - Listing has been approved by admin
- `listing_rejected` - Listing has been rejected by admin
- `listing_expiring` - Listing is about to expire
- `document_expiring` - Company document is about to expire
- `account_verified` - Company account has been verified
- `account_rejected` - Company account verification rejected

## Metadata Structure

The metadata field contains JSON data specific to each notification type:

### Offer Notifications
```json
{
  "listingId": 456,
  "offerId": 789,
  "offerAmount": 1000.00,
  "currency": "GBP"
}
```

### Listing Notifications
```json
{
  "listingId": 456,
  "listingTitle": "HDPE Bottles - High Quality",
  "rejectionReason": "Insufficient documentation"
}
```

### Document Notifications
```json
{
  "documentId": 123,
  "documentType": "environmental_permit",
  "expiryDate": "2025-06-01T00:00:00.000Z"
}
```

## Authentication

All notification endpoints require JWT authentication. Include the token in the Authorization header:

```
Authorization: Bearer YOUR_JWT_TOKEN
```

## Best Practices

1. **Mark as Read**: Always update `isRead` and `readAt` when user views notification
2. **Pagination**: Use limit/skip for large notification lists
3. **Filtering**: Filter by `userId` and `isRead` for user-specific unread notifications
4. **Metadata**: Store relevant IDs and data in metadata for navigation purposes
5. **Cleanup**: Implement periodic cleanup of old read notifications
